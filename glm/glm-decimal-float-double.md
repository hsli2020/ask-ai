# 我看到一种说法，在数据库中存储金额之类的数据时，必须用DECIMAL类型，不能用FLOAT或DOUBLE类型，请解释一下这样做的原因，以及不这样做可能会造成的后果，最好给出例子

这是一个非常经典且至关重要的问题。那个说法是**完全正确**的。

简单来说：**FLOAT 和 DOUBLE 是“近似值”，而 DECIMAL 是“精确值”。**

在涉及金钱、财务、支付等场景时，数据的**精确性**高于一切，容不得半点误差。下面我将从原理、后果和示例三个方面为您详细解释。

### 1. 核心原因：计算机是如何存储小数的？

要理解这个问题，首先要明白计算机底层存储数据的机制。

*   **FLOAT / DOUBLE (浮点数)：**
    它们使用二进制浮点数算术标准（IEEE 754）。计算机是二进制的（0和1），它只能精确表示分母为 2 的幂次方的分数（如 1/2, 1/4, 1/8）。
    然而，很多我们常见的十进制小数（如 0.1, 0.2, 0.3）在二进制中是**无限循环小数**。
    就像我们在十进制中无法精确表示 1/3（0.33333...）一样，二进制也无法精确表示 0.1。
    因此，FLOAT 和 DOUBLE 存储的是**近似值**，会有微小的舍入误差。

*   **DECIMAL (定点数)：**
    它将数据分为整数部分和小数部分分开存储，或者以一种精确的格式存储。它不是存储近似值，而是存储**该数值的精确表示**。只要你定义的精度范围内（例如 `DECIMAL(10, 2)`），它就能分毫不差地存储数据。

### 2. 不这样做的后果

如果在金额字段使用了 FLOAT 或 DOUBLE，可能会导致以下严重后果：

1.  **账对不上（资金损失）：** 系统显示用户余额为 0.00，但数据库判断它大于 0，导致用户能提现 0.01 元，或者反之，导致资金凭空消失。
2.  **累计误差：** 单次误差可能极小（小数点后第 16 位），但在进行大量的累加、乘除运算（如银行利息计算、大规模订单汇总）后，误差会迅速放大，导致最终结果严重偏离。
3.  **查询失效：** 明明看起来相等的数据，用 SQL 查询 `WHERE amount = 0.3` 却查不出来，因为数据库里存的实际是 `0.30000000000000004`。
4.  **法律风险：** 财务数据必须精准，任何审计发现账目不平都是严重的事故。

### 3. 具体的例子

为了直观展示，我们可以通过 SQL 或者编程语言来演示这个现象。

#### 场景一：简单的加法运算

假设我们有一个简单的需求：0.1 元 + 0.2 元。

**如果是 DOUBLE 类型（二进制浮点）：**
计算机内部的计算过程大约是这样的：
*   0.1 在二进制中是无限循环小数。
*   0.2 在二进制中也是无限循环小数。
*   相加后，结果不是 0.3，而是 **0.30000000000000004**。

**验证代码（以 MySQL 为例）：**

```sql
-- 使用 DOUBLE
SELECT 0.1 + 0.2; 
-- 结果可能显示为 0.3（因为客户端显示工具可能会自动四舍五入），但实际内存值并非如此

-- 让我们强制显示更多位数
SELECT FORMAT(0.1 + 0.2, 30); 
-- 结果: 0.300000000000000044408920985006
```

你看，结果并不是纯粹的 0.3。

**如果是 DECIMAL 类型：**

```sql
-- 使用 DECIMAL
SELECT FORMAT(CAST(0.1 AS DECIMAL(10,2)) + CAST(0.2 AS DECIMAL(10,2)), 30);
-- 结果: 0.300000000000000000000000000000
```
这才是完美的 0.3。

#### 场景二：比较查询失败

这是一个非常容易踩的坑。假设数据库里存了某商品价格为 `0.3` 元（用的是 FLOAT）。

```sql
-- 模拟建表
CREATE TABLE price_test (
    id INT PRIMARY KEY,
    price FLOAT
);

INSERT INTO price_test (id, price) VALUES (1, 0.3);

-- 尝试查询价格为 0.3 的商品
SELECT * FROM price_test WHERE price = 0.3;
```
在某些数据库系统或特定配置下，这条 SQL 可能返回**空结果**，或者无法返回预期行。为什么？
因为存进去的时候，0.3 变成了 `0.300000011920928...`，而你查询条件里的 0.3 被解析为另一个近似值。这两个近似值不相等！

**正确的做法：**
如果使用 DECIMAL，`WHERE price = 0.30` 永远能精准命中。

#### 场景三：累加造成的“消失的钱”

假设你经营一个平台，有一笔订单金额是 0.01 元。

*   **FLOAT:** 0.01 在二进制中无法精确表示。
*   如果你有一万笔这样的订单，累加起来的总金额可能不是 100.00 元，而是 `99.999999...` 或者 `100.000001...`。
*   当你要给商户结算时，如果你用 `FLOOR()` 函数（向下取整）来计算应打款金额：
    *   正确结果：100.00 元。
    *   FLOAT 结果：`FLOOR(99.999999)` = 99.00 元。
    *   **结果：平台凭空“赚”了 1 元钱，商户损失了 1 元钱。**

### 4. 最佳实践建议

1.  **数据库设计：**
    金额字段**必须**使用 `DECIMAL` 类型。
    语法：`DECIMAL(M, D)`
    *   `M`：总位数（精度）。
    *   `D`：小数位数。
    *   例如金额通常是 `DECIMAL(10, 2)` 或者 `DECIMAL(18, 4)`（如果需要精确到厘或毫秒级精度）。

2.  **另一种方案：存为整数**
    在某些高性能或遗留系统中，金额会以**“分”**为单位存储为 `BIGINT` 类型。
    *   例如：1.23 元存为 123。
    *   这样完全规避了小数问题，运算速度极快，且绝对精确。
    *   显示时只需除以 100 即可。

**总结：**
永远不要相信浮点数能精确表示金钱。**在钱的问题上，要么用 DECIMAL，要么用整数。**
