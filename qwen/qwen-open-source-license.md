# 现在最流行的开源协议有哪些，总结一下它们各自的要求，适用场合。

# 📋 2026年主流开源协议对比指南

开源许可证主要分为两大类：**宽松式（Permissive）** 和 **著佐权（Copyleft）**[[5]]。以下是目前最流行的开源协议详解：

---

## 🔹 宽松式许可证（可闭源商用）

### 1. MIT License（最宽松）
| 项目 | 内容 |
|------|------|
| ✅ 允许 | 商业使用、修改、分发、再授权、私有使用 |
| ❌ 限制 | 作者不承担任何责任 |
| 📝 必须 | 保留原始版权声明和许可证全文 |
| 🎯 适用场景 | 个人项目、初创公司、希望最大化传播的代码、教学示例 |
| 🌰 知名项目 | React、Vue.js、Node.js、jQuery |

> MIT 是"极简主义的代表"，唯一要求是保留版权声明即可[[1]][[9]]。

---

### 2. Apache License 2.0（企业首选）
| 项目 | 内容 |
|------|------|
| ✅ 允许 | 商业使用、修改、分发、再授权、专利使用、提供担保 |
| ❌ 限制 | 不能用贡献者商标推广、作者不担责 |
| 📝 必须 | 保留版权/许可证声明、注明修改内容、包含 NOTICE 文件 |
| 🔐 特色 | **明确专利授权条款**，防止贡献者事后提起专利诉讼[[3]] |
| 🎯 适用场景 | 企业级开源项目、需规避专利风险、希望商业友好的项目 |
| 🌰 知名项目 | Android、Kubernetes、Hadoop、TensorFlow |

---

### 3. BSD License（学术友好）
| 类型 | 核心区别 |
|------|----------|
| BSD-2-Clause | 与 MIT 基本相同，保留版权声明即可 |
| BSD-3-Clause | 额外禁止**用原作者名义推广衍生产品**[[10]] |

| ✅ 允许 | 商业使用、修改、分发、闭源 |
| 📝 必须 | 保留版权声明、免责声明 |
| 🎯 适用场景 | 学术界、操作系统生态、不希望被"借名营销"的项目 |
| 🌰 知名项目 | FreeBSD、Nginx、Go 语言早期版本 |

---

## 🔹 著佐权许可证（强制开源）

### 4. GPL v2/v3（强传染性）
| 项目 | 内容 |
|------|------|
| ✅ 允许 | 商业使用、修改、分发 |
| ❌ 限制 | **衍生作品必须同样以 GPL 开源**（"传染性"）[[12]] |
| 📝 必须 | 公开源码、保留版权、注明修改、包含许可证 |
| ⚠️ v3 新增 | 禁止硬件锁定（Tivoization）、明确专利条款 |
| 🎯 适用场景 | 希望确保代码永远开源、技术民主化项目 |
| 🌰 知名项目 | Linux 内核(GPLv2)、Git、WordPress |

> ⚠️ 风险：使用 GPL 库可能导致整个商业软件被迫开源[[17]]

---

### 5. LGPL v2.1/v3（库的折中方案）
| 项目 | 内容 |
|------|------|
| ✅ 允许 | 专有软件**动态链接**LGPL库而不必开源[[13]] |
| ❌ 限制 | 修改LGPL库本身则修改部分必须开源 |
| 🎯 适用场景 | 开源库开发者，希望被商业软件广泛调用 |
| 🌰 知名项目 | GTK、FFmpeg 部分组件 |

---

### 6. AGPL v3（云服务专用）
| 项目 | 内容 |
|------|------|
| 🔍 核心 | 填补 GPL 的"网络漏洞"：通过网络提供服务也**必须提供源码**[[27]] |
| 🎯 适用场景 | SaaS 服务、云原生应用、防止云厂商"白嫖" |
| 🌰 知名项目 | Nextcloud、MongoDB（曾使用）、Mattermost |

---

### 7. MPL 2.0（文件级 Copyleft）
| 项目 | 内容 |
|------|------|
| 🔍 核心 | **仅修改过的文件需开源**，新增文件可闭源[[24]] |
| ✅ 优势 | 与专有代码链接不触发整体开源义务 |
| 🎯 适用场景 | 平衡商业友好与开源要求的中间路线 |
| 🌰 知名项目 | Firefox、Thunderbird |

---

## 🧭 快速选择决策树

```
你的项目目标是什么？
│
├─ 🚀 最大化传播，不介意被闭源商用 → MIT
│
├─ 🏢 企业项目，需要专利保护 → Apache 2.0
│
├─ 📚 学术/系统项目，不想被借名 → BSD-3-Clause
│
├─ 🔐 确保衍生作品永远开源 → GPL v3
│
├─ 📦 开发库，希望被商业软件调用 → LGPL 或 MPL 2.0
│
├─ ☁️ 提供 SaaS 服务，防止云厂商白嫖 → AGPL v3
│
└─ ⚖️ 既要开源又要商业友好 → MPL 2.0 或 Apache 2.0
```

---

## ⚠️ 关键注意事项

1. **许可证兼容性**：GPL v2 与 Apache 2.0 不兼容，但 GPL v3 兼容[[11]]
2. **依赖传递**：使用 npm/pip 安装依赖时，需检查依赖树的许可证[[1]]
3. **无许可证 ≠ 开源**：未声明 LICENSE 文件意味着代码受版权保护，他人无权使用[[1]]
4. **近期趋势**：部分项目（如 Redis、ElasticSearch）因商业考量变更为"伪开源"许可证（如 SSPL），OSI 明确表示此类许可证**不符合开源定义**[[6]]

## 📊 协议对比速查表

| 协议 | 可闭源商用 | 需开源衍生 | 专利保护 | 传染性 | 商业友好度 |
|------|-----------|-----------|----------|--------|-----------|
| MIT | ✅ | ❌ | ❌ | 无 | ⭐⭐⭐⭐⭐ |
| BSD | ✅ | ❌ | ❌ | 无 | ⭐⭐⭐⭐⭐ |
| Apache 2.0 | ✅ | ❌ | ✅ | 无 | ⭐⭐⭐⭐⭐ |
| MPL 2.0 | ⚠️文件级 | ⚠️文件级 | ✅ | 弱 | ⭐⭐⭐⭐ |
| LGPL | ⚠️动态链接 | ✅修改部分 | ✅ | 中 | ⭐⭐⭐ |
| GPL | ❌ | ✅整体 | ⚠️ | 强 | ⭐⭐ |
| AGPL | ❌ | ✅+网络服务 | ⚠️ | 最强 | ⭐ |

> 💡 **建议**：使用 [choosealicense.com](https://choosealicense.com)（GitHub 官方）或 [tldrlegal.com](https://tldrlegal.com) 辅助决策[[1]]。

选择开源协议本质是选择你对"自由"的定义：是**用户的使用自由**（宽松协议），还是**代码的开放自由**（Copyleft 协议）[[1]]。根据你的项目目标、商业模式和社区策略做出明智选择！
