# 传统的web应用都是“请求/应答”模式，客户端先发起请求，服务器端发送应答。但某些应用类型，比如股市行情网站，需要服务器主动发送变化，避免客户端定期轮询方式。请介绍一下有哪些技术方案可实现服务器端主动向客户端发送信息。

为了实现服务器端主动向客户端发送信息（即“服务器推送”或“实时通信”），避免传统轮询（Polling）带来的高延迟和服务器资源浪费，业界发展出了多种技术方案。

针对您提到的**股市行情网站**这类需要高频、低延迟单向或双向数据同步的场景，以下是目前主流的技术方案及其特点：

---

### 1. Server-Sent Events (SSE)
SSE 是一种基于 HTTP 协议的**单向**通信技术，专门用于服务器向客户端推送事件流。
* **工作原理**：客户端通过 `EventSource` API 发起一个普通的 HTTP 请求，服务器保持连接打开，并持续以 `text/event-stream` 的 MIME 类型发送数据。
* **优点**：
  * **轻量且简单**：基于标准 HTTP，无需像 WebSocket 那样进行协议升级，容易穿透防火墙和代理。
 * **原生支持重连**：浏览器内置了自动重连机制。
  * **支持断点续传**：通过 `Last-Event-ID` 机制，客户端断线重连时，服务器可以知道上次发送到哪，避免行情数据遗漏。
* **缺点**：只能单向通信（服务器 → 客户端），且受限于浏览器的并发连接数限制（通常每个域名最多 6 个并发 SSE 连接）。
* **适用场景**：**股市行情推送**、新闻滚动、日志监控等**单向**数据流场景。

### 2. WebSocket
WebSocket 是目前最成熟的**全双工**实时通信协议。
* **工作原理**：客户端通过 HTTP 发起一个带有 `Upgrade: websocket` 头的请求，服务器同意后，连接从 HTTP 升级为 WebSocket 协议（基于 TCP）。此后，双方可以在同一个持久连接上自由、双向地发送数据帧。
* **优点**：
  * **真正的双向实时通信**：延迟极低，头部开销极小（仅 2~10 字节）。
  * **无同源限制**（配合 CORS 或特定配置）。
* **缺点**：
  * 需要服务器端有专门的 WebSocket 支持（如 Node.js, Go, Java Netty 等）。
  * 负载均衡配置较复杂（通常需要 Sticky Session 或基于 Redis 的共享会话状态）。
  * 需要自行实现心跳保活（Ping/Pong）和断线重连逻辑。
* **适用场景**：高频交易行情（毫秒级 Tick 数据）、在线聊天、多人协作编辑、实时游戏。

### 3. HTTP Long Polling (长轮询)
这是早期在 WebSocket 普及前最常用的折中方案。
* **工作原理**：客户端发起 HTTP 请求，服务器如果没有新数据，会**挂起（Hold）** 该请求而不是立即返回；直到有数据更新或超时，服务器才返回响应。客户端收到响应后，**立即**发起下一次请求。
* **优点**：兼容性极佳，几乎支持所有浏览器和网络环境；实现相对简单。
* **缺点**：每次请求仍带有完整的 HTTP Header，带宽浪费较大；服务器需要维持大量挂起的连接，消耗较多内存和文件描述符。
* **适用场景**：作为 SSE 或 WebSocket 的**降级兜底方案**，或客户端环境极度受限的场景。

### 4. WebTransport (下一代技术)
基于 HTTP/3 (QUIC) 的新兴 Web API，旨在提供比 WebSocket 更低延迟、更灵活的传输。
* **工作原理**：建立在 QUIC 协议之上，支持多路复用，且同时支持**可靠传输**（类似 TCP）和**不可靠传输**（类似 UDP 的 Datagram）。
* **优点**：
  * **无队头阻塞**：基于 UDP，网络波动时不会像 TCP/WebSocket 那样导致整体卡顿。
  * **连接建立更快**：0-RTT 或 1-RTT 握手。
  * 支持双向流和数据报，非常适合对延迟极度敏感且允许少量丢包的场景。
* **缺点**：浏览器支持仍在普及中（Chrome/Edge 已稳定支持，Safari/Firefox 正在跟进），服务端生态（如基于 QUIC 的服务器）不如 WebSocket 成熟。
* **适用场景**：超低延迟的实时游戏、云游戏、极高并发的实时音视频或金融高频数据流。

---

### ⚠️ 避坑提示：HTTP/2 Server Push
虽然 HTTP/2 规范中定义了 Server Push，但它**不适合**用于推送动态业务数据（如股市行情）。它的设计初衷是让服务器在客户端请求 HTML 时，主动推送关联的静态资源（如 CSS、JS、图片）。目前，Chrome 等主流浏览器已**正式废弃并移除**了对 HTTP/2 Server Push 的支持。

### 方案对比与选型建议（以股市行情为例）

| 特性 | SSE | WebSocket | 长轮询 | WebTransport |
| :--- | :--- | :--- | :--- | :--- |
| **通信方向** | 单向 (Server → Client) | 双向 | 伪双向 | 双向 |
| **协议基础** | HTTP/1.1 或 HTTP/2 | TCP (ws/wss) | HTTP | HTTP/3 (QUIC/UDP) |
| **延迟** | 低 | 极低 | 中高 | 极低 |
| **浏览器支持** | 极好 (除极老旧浏览器) | 极好 | 极好 | 逐步普及中 |
| **断线重连** | 原生自动支持 | 需手动代码实现 | 原生机制 | 需手动实现 |
| **开发复杂度** | 低 | 中 | 低 | 高 |

#### 针对“股市行情网站”的选型建议：
1. **首选 SSE**：如果您的需求仅仅是**服务器向客户端单向推送**最新的股票价格、K线数据，SSE 是最佳选择。它开发成本最低，且原生的 `Last-Event-ID` 机制能完美解决网络抖动导致的行情漏发问题。
2. **选用 WebSocket**：如果行情更新频率极高（如毫秒级逐笔交易数据），或者客户端需要频繁地向服务器发送复杂的订阅/取消订阅指令（双向交互频繁），WebSocket 是目前的行业标准。
3. **降级策略**：在实际工程中，通常会采用“能力检测”策略：优先尝试建立 WebSocket/SSE 连接，如果失败（如被企业防火墙拦截），则自动降级为长轮询，以保证服务的可用性。

### 工程实践注意事项
无论选择哪种长连接技术，在生产环境中都必须考虑以下几点：
1. **心跳机制 (Heartbeat)**：定期发送 Ping/Pong 消息，防止中间代理（如 Nginx、云负载均衡器）因空闲超时而静默断开连接。
2. **断线重连与指数退避**：客户端断线后应自动重连，且重连间隔应采用指数退避算法（如 1s, 2s, 4s, 8s），避免服务器在恢复瞬间被海量重连请求击垮（雪崩效应）。
3. **状态共享**：如果使用 WebSocket 且部署了多台服务器，用户的连接可能落在不同机器上。需要通过 Redis 或消息队列（如 Kafka/RabbitMQ）来共享订阅状态和路由消息。
