# 关于https的一个疑问，我访问一个https的网站，url中带有很多参数，其中可能会有一些敏感信息，这些信息会被暴露出去吗？

这是一个非常经典且重要的安全问题。简单直接的回答是：

**在传输过程中（从你的浏览器到服务器），URL 中的参数是加密的，第三方（如 ISP、黑客、路由器）无法直接看到。但是，这些信息会在其他地方（如服务器日志、浏览器历史、Referer 头等）留下痕迹，存在暴露风险。**

为了让你完全理解，我们需要把“暴露”的场景拆解来看：

### 1. 网络传输过程中（相对安全）
当你在浏览器输入 `https://example.com/page?token=123&password=abc` 并回车时：
*   **加密范围：** HTTPS (TLS/SSL) 加密的是建立连接之后的**所有 HTTP 报文**。这包括 HTTP 请求行（包含 URL 路径和参数）、请求头（Headers）和请求体（Body）。
*   **谁能看到：** 你的互联网服务提供商（ISP）、公共 WiFi 的运营商、或者中间的路由器，他们只能看到你连接了 `example.com` 这个域名（通过 SNI 和 DNS），以及你的 IP 地址。**他们无法看到你 URL 中 `?` 后面的具体参数内容。**
*   **例外情况：** 如果是在公司内网或特定网络环境下，机构安装了受信任的根证书并进行“中间人代理（MITM）”监控，那么他们是可以解密并看到完整 URL 的。

### 2. 容易暴露的“盲区”（高风险）
虽然传输加密了，但敏感信息放在 URL 参数里（即使用 GET 请求）在以下环节极易泄露：

#### A. 服务器访问日志 (Server Access Logs)
*   **风险：** Web 服务器（如 Nginx, Apache, IIS）默认会将完整的请求 URL（包含参数）记录在访问日志中。
*   **后果：** 如果服务器被黑客入侵，或者运维人员权限管理不当，日志中的敏感信息（如 Token、手机号、身份证）就会泄露。
*   **现状：** 这是 URL 带参最大的安全隐患。

#### B. 浏览器历史记录 (Browser History)
*   **风险：** 浏览器会将你访问过的完整 URL 保存在本地历史记录中。
*   **后果：** 任何能访问你电脑或手机的人，打开浏览器历史记录，就能看到带有敏感参数的 URL。如果浏览器开启了账号同步，这些记录甚至可能上传到云端。

#### C. Referer 头 (引荐来源)
*   **风险：** 如果你在 `https://site-A.com/page?secret=123` 页面上点击了一个链接跳转到 `https://site-B.com`，浏览器通常会在发给 B 网站的请求头中带上 `Referer: https://site-A.com/page?secret=123`。
*   **后果：** 第三方网站（site-B）就能获取到你上一页面的敏感参数。
*   **注意：** 即使是 HTTPS 跳 HTTPS，Referer 默认也是会带的（除非设置了 `Referrer-Policy`）。

#### D. 代理服务器和网关日志
*   **风险：** 公司网关、学校防火墙或某些 CDN 节点可能会记录完整的 URL 用于审计或缓存优化。

#### E. 域名可见性 (SNI)
*   **风险：** 虽然参数加密了，但在 TLS 握手阶段，传统的 **SNI (Server Name Indication)** 是明文的。
*   **后果：** 外界虽然不知道你的具体参数，但知道你访问了哪个域名。

### 3. 最佳实践建议

基于以上风险，**业界的安全共识是：永远不要把敏感信息放在 URL 参数中。**

如果你正在开发网站或设计 API，请遵循以下原则：

1.  **敏感数据用 POST Body：**
    *   登录密码、身份证号、银行卡号、Access Token 等，应该放在 HTTP 请求的 **Body** 中（使用 POST/PUT 方法），而不是 URL 参数。
    *   Body 不会被记录在服务器访问日志的默认配置中（通常日志只记 URL），也不会出现在浏览器历史记录里。

2.  **使用 Authorization Header：**
    *   身份认证令牌（Token）最好放在 HTTP 请求头中，例如 `Authorization: Bearer <token>`。
    *   请求头通常比 URL 更不容易被意外记录。

3.  **如果必须用 GET 参数：**
    *   确保参数是非敏感的（如搜索关键词、分页页码）。
    *   如果必须传递临时凭证，确保其**有效期极短**且**一次性使用**。
    *   配置服务器日志脱敏（例如配置 Nginx 不记录特定参数的值）。
    *   设置 HTTP 头 `Referrer-Policy: no-referrer` 或 `strict-origin-when-cross-origin`，防止泄露给第三方。

4.  **前端跳转注意：**
    *   在敏感页面避免放置指向外部网站的链接，防止 Referer 泄露。

### 总结
*   **黑客在网路上窃听：** 看不到你的 URL 参数（HTTPS 保护）。
*   **服务器管理员/日志泄露：** **看得到**（高风险）。
*   **借用你电脑的人：** **看得到**（浏览器历史）。
*   **你跳转到的第三方网站：** **可能看得到**（Referer）。

**结论：** 为了安全起见，**请勿在 URL 参数中传递任何敏感信息**，无论是否使用了 HTTPS。
